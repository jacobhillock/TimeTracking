# 009.3 - Jira Automation

## Goal

Automatically send the one `TimeLogSummary` for a given client, ticket, and day to Jira when it is created or updated, using a debounced background flow so repeated entry edits do not spam Jira.

## Assumptions

1. Jira automation runs only after auth verification succeeds.
2. There is exactly one summary per client, ticket, and day, and that record is the sync unit.
3. A summary may be created before Jira has assigned an issue/log identifier, so the local summary must remain the source of truth until Jira responds.
4. Repeated edits to the same summary should collapse into one outbound Jira update.

## Implementation Plan

### 1. Expand the Jira service layer

- Add request helpers in `src/services/jira.ts` for:
  - creating a Jira log entry
  - updating an existing Jira log entry
  - mapping Jira responses back to local summary fields
- Keep the Jira request and response shapes isolated from the rest of the app.

### 2. Add a sync state for summaries

- Extend `TimeLogSummary` with the fields needed for automation:
  - `jiraId`
  - `logged`
  - any sync metadata needed to prevent duplicate sends
- Distinguish between:
  - summary exists locally
  - summary has been sent to Jira
  - summary needs another update because a user edited an entry again
- Treat summary updates as full replacements for the client-ticket-day record, not incremental per-entry logs.

### 3. Build a debounced sync queue

- Add a small queue or scheduler in the service layer, not in React components.
- Debounce outbound sync by summary key so a rapid series of edits coalesces into one Jira request.
- Flush the latest state when the app is idle long enough or when the user finishes editing.
- Prevent concurrent updates for the same summary key from racing each other.

### 4. Trigger sync from summary writes

- After a summary is created or updated locally:
  - if auth is verified, enqueue a Jira sync
  - if auth is not verified, keep the summary pending and do not send
- On successful Jira response:
  - store `jiraId`
  - mark the summary as logged
  - clear any pending sync marker

### 5. Handle retries and failure states

- Preserve the local summary if Jira fails.
- Record enough failure state to retry later without losing the latest edits.
- Avoid marking a summary as logged until Jira confirms success.

### 6. Update the UI to reflect sync state

- Show whether a summary is pending, synced, or failed to sync.
- Keep the display lightweight; the main workflow should still work even if Jira is unreachable.

### 7. Add cleanup and migration support

- Backfill any existing summaries when the feature ships so old data can sync on the next edit.
- Ensure summaries that already have a Jira ID do not get recreated.

## Files Likely To Change

- `src/services/jira.ts`
- `src/services/timeEntryService.ts`
- `src/services/db.ts`
- `src/App.tsx`
- `src/components/TaskView.tsx` if it renders summary state
- `src/services/types.ts`

## Verification

1. Edit the same entry multiple times quickly and confirm Jira receives one debounced update.
2. Confirm `jiraId` is stored after a successful create.
3. Confirm failed Jira calls do not lose the local summary.
4. Confirm the app does not resend already-synced summaries on reload.
