# 009.2 - Jira Auth

## Goal

Collect a Jira auth token in settings, verify it against Jira, and expose a non-editable verified state so later automation can trust the token before sending logs.

## Assumptions

1. `jiraBaseUrl` already exists and will continue to be the base for API calls.
2. The app will keep using local persistent settings unless the project later introduces a dedicated secure storage layer.
3. Verification should happen against Jira immediately after token entry, and the verified flag should clear when the token changes.

## Implementation Plan

### 1. Extend settings state

- Add `jiraAuthToken` to the settings context and local-storage-backed settings hook.
- Add `jiraAuthTokenVerified` as a derived or stored boolean setting.
- Keep `jiraAuthTokenVerified` non-editable in the UI.

### 2. Add the settings UI

- Update the settings section in `src/App.tsx` to include:
  - a token input field
  - a verify status indicator
  - a short helper message explaining what verification means
- Clear the verified flag when the token input changes.
- Avoid leaking the token back into the interface once it has been entered, beyond what the browser input normally shows.

### 3. Add Jira verification logic

- Expand `src/services/jira.ts` into a real Jira API helper module.
- Implement a small verification call that proves the token works against the configured Jira base URL.
- Prefer a minimal request that is cheap and deterministic, such as fetching the current user or another endpoint Jira accepts for auth validation.
- Return structured success/failure results so the UI can show a clear message.
- Send Jira credentials in an `Authorization` header using Basic auth with the token.

### 4. Decide on HTTP client strategy

- Start with `fetch` unless Jira-specific request ergonomics or project conventions make axios clearly better.
- Only add axios if it materially reduces implementation risk or complexity.
- Keep request logic isolated so later automation can reuse the same client.

### 5. Wire verification into state transitions

- When the user saves or changes the token:
  - set verified to false immediately
  - trigger verification
  - on success, set verified to true
  - on failure, keep verified false and surface the error
- Guard against stale responses if the user edits the token multiple times quickly.

### 6. Reuse verified auth in future Jira calls

- Centralize the auth header construction in one helper.
- Format the header as `Authorization: Basic <base64(user:token)>` according to Jira's token auth expectations.
- Make later Jira-logging code depend on the verified token state instead of raw user input alone.

## Files Likely To Change

- `src/context/SettingsContext.tsx`
- `src/hooks/useLocalStorageState.ts`
- `src/services/jira.ts`
- `src/App.tsx`
- `src/types/app.ts` if the settings type needs to expand

## Verification

1. Enter a valid token and confirm verified state becomes true.
2. Enter an invalid token and confirm the UI reports failure.
3. Change the token and confirm verified state resets.
4. Reload the page and confirm the stored token and verified state behave as expected.
