# 009.1 - Time Log Summary Consolidation

## Goal

Move ticket-level logging from per-entry descriptions into a dedicated `TimeLogSummary` record so Jira-ready data has one source of truth per client/ticket/day.

## Assumptions

1. `TimeEntry` remains the UI editing unit for now, but summary data becomes the canonical place for the aggregated log message and logged state.
2. The new summary store is keyed by `client-ticket-yyyy-MM-dd` and must be queryable by date and ticket pair.
3. Existing entries should be migrated without losing user data.
4. There must only be one summary per client, ticket, and day.

## Implementation Plan

### 1. Extend the data model

- Add a new `TimeLogSummary` type in the shared types layer.
- Include the minimum fields required by the requirement:
  - `id`
  - `client`
  - `ticket`
  - `date`
  - `description`
  - `logged`
  - `jiraId` for later phases
  - a stored total-time field if the view needs the precomputed duration
- Keep the type specific to this app instead of importing the shared `oil-blender` model directly.

### 2. Add a new IndexedDB store

- Update `src/services/db.ts` to add a new object store for summaries.
- Use a key path on `id` so upserts are straightforward.
- Add indexes for the read patterns we need:
  - by `date`
  - by `client`
  - by `ticket`
  - optionally a composite-like derived field if the summary view needs grouped lookups
- Bump the DB version and add migration logic that creates the store safely when upgrading existing users.

### 3. Build summary CRUD helpers

- Reuse the CRUD helper from `../oil-blender/src/utilities/idb.ts` as the database access layer for this feature.
- Replace the current `src/services/db.ts` implementation with that helper pattern instead of layering a second DB abstraction on top.
- Adapt it to this app's types and indexes instead of rewriting CRUD calls by hand.
- Update the app's shared types and any DB/config constants so they match the new summary store shape and indexes.
- Add a thin service module for summary operations, or extend the existing time entry service if that is cleaner.
- Implement helpers for:
  - `getSummaryForEntry(date, client, ticket)`
  - `upsertSummaryFromEntry(entry, date)`
  - `updateSummaryTotals(date, client, ticket, duration, description)`
  - `getSummariesForDate(date)`
- Keep the DB access behind service helpers instead of reaching into `db.ts` from components.

### 4. Sync entries into summaries

- When an entry is created or edited:
  - look up the matching summary by `client-ticket-date`
  - create it if missing
  - set or refresh the summary description from the entry description
  - recalculate total logged time for that ticket/day
- Ensure the summary update happens whenever an entry is created, assigned, or edited.

### 5. Migrate existing data

- On startup or DB upgrade, scan existing entries and seed summaries from them.
- Preserve the most useful description when multiple entries map to the same summary key.
- Mark a summary as logged when all relevant entries for that client/ticket/day are logged.
- If the underlying entries are not all logged, the summary must remain unlogged until that changes.

### 6. Remove redundant entry fields only after callers are moved

- Once summary reads are in place, remove any now-redundant entry-level fields from the UI and services.
- Keep the migration path conservative so old data still renders correctly while the refactor is in progress.

### 7. Update summary UI consumers

- Update the summary panel to read from `TimeLogSummary` data instead of reconstructing everything from raw entries.
- Keep the existing ticket/client grouping behavior intact while changing the source of the description and total time.

## Files Likely To Change

- `src/services/db.ts`
- `src/services/types.ts`
- `src/services/timeEntryService.ts`
- `src/services/searchService.ts`
- `src/App.tsx`

2. Confirm the summary total updates when entry times change.
3. Confirm existing entries migrate without data loss.
4. Confirm the UI still renders when there are no summaries yet.
